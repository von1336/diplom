# Сервис Автоматизации Закупок

Backend-приложение на Django REST Framework для взаимодействия покупателей, поставщиков и администратора в единой системе закупок. Проект решает типовой сценарий B2B/B2C-каталога: поставщики публикуют и обновляют прайс-листы, покупатели формируют корзину из товаров разных магазинов, оформляют заказ и получают подтверждение, а поставщики и администратор работают с поступившими заказами.

## Назначение системы

Сервис объединяет несколько поставщиков в общий каталог и предоставляет единый API для следующих задач:

- публикация и обновление ассортимента поставщика из YAML-прайса;
- просмотр общего каталога товаров с фильтрацией по магазину и категории;
- управление корзиной и контактами покупателя;
- оформление заказа на доставку;
- уведомление участников процесса по email;
- администрирование каталога и заказов через Django Admin.

Проект реализован как API-first backend. Основной сценарий взаимодействия проходит через REST endpoint, а административные операции доступны через встроенную админку Django.

## Роли в системе

### Покупатель

Покупатель регистрируется, подтверждает email, проходит авторизацию и получает token для работы с API. После этого он может:

- просматривать каталог;
- добавлять позиции в корзину;
- управлять адресами доставки;
- подтверждать заказ;
- просматривать историю своих заказов.

### Поставщик

Поставщик работает со своей витриной и заказами, относящимися только к его ассортименту. Для него доступны:

- загрузка прайса из YAML-файла по URL;
- включение и выключение магазина для приема заказов;
- просмотр заказов, содержащих его товарные позиции.

### Администратор

Администратор работает через Django Admin и может:

- просматривать пользователей, магазины, категории, товары и параметры;
- контролировать заказы и их состояние;
- запускать импорт товаров через отдельную страницу в админке.

## Бизнес-потоки

### Регистрация и подтверждение пользователя

После регистрации создается неактивный пользователь и токен подтверждения email. До подтверждения адреса вход в систему недоступен. После подтверждения пользователь может авторизоваться и получить token для дальнейших запросов к API.

### Публикация каталога поставщика

Поставщик передает ссылку на YAML-файл. Система загружает прайс, создает или обновляет магазин, категории, товары и характеристики, а затем публикует позиции в общем каталоге. Товары доступны в выдаче только у активных магазинов.

### Работа с корзиной и заказом

Корзина хранится как заказ в состоянии `basket`. Пользователь может добавлять позиции, менять количество и удалять их. После выбора контактных данных корзина переводится в состояние `new` и становится оформленным заказом.

### Email-уведомления

Система отправляет письма в трех ключевых точках:

- подтверждение регистрации;
- сброс пароля;
- уведомление о создании заказа.

Если Redis и Celery доступны, задачи отправляются в фон. Если брокер недоступен, приложение не падает: импорт и отправка писем выполняются синхронно в Django-процессе.

## Предметная модель

В проекте реализована следующая модель данных:

- `User` — пользователь системы с ролью `buyer` или `shop`;
- `Shop` — магазин поставщика;
- `Category` — категория товаров;
- `Product` — товарная сущность;
- `ProductInfo` — предложение конкретного магазина по товару;
- `Parameter` и `ProductParameter` — характеристики товара;
- `Contact` — адрес и контактные данные покупателя;
- `Order` — заказ пользователя;
- `OrderItem` — позиция заказа;
- `ConfirmEmailToken` — токен подтверждения email.

Такое разделение позволяет хранить общий товар и несколько предложений от разных поставщиков отдельно, что удобно для каталога, корзины и заказов.

## Ключевой функционал

### Для покупателей

- регистрация и подтверждение email;
- token-аутентификация;
- просмотр профиля;
- просмотр каталога, категорий и магазинов;
- управление контактами доставки;
- управление корзиной;
- оформление и просмотр заказов;
- сброс пароля.

### Для поставщиков

- регистрация с ролью `shop`;
- импорт ассортимента из YAML;
- управление статусом магазина;
- просмотр заказов по своему ассортименту.

### Для администратора

- доступ к модели пользователей, магазинов, товаров и заказов;
- изменение статусов заказов в админке;
- запуск импорта из отдельной admin-страницы;
- просмотр токенов подтверждения и контактных данных.

## REST API

Базовый префикс для всех endpoint:

```text
/api/v1/
```

### Пользовательский контур

- `POST /api/v1/user/register`
- `POST /api/v1/user/register/confirm`
- `POST /api/v1/user/login`
- `GET /api/v1/user/details`
- `POST /api/v1/user/details`
- `GET /api/v1/user/contact`
- `POST /api/v1/user/contact`
- `PUT /api/v1/user/contact`
- `DELETE /api/v1/user/contact`
- `POST /api/v1/user/password_reset`
- `POST /api/v1/user/password_reset/confirm`

### Каталог

- `GET /api/v1/categories`
- `GET /api/v1/shops`
- `GET /api/v1/products`
- `GET /api/v1/products?shop_id=<id>`
- `GET /api/v1/products?category_id=<id>`

### Корзина и заказ

- `GET /api/v1/basket`
- `POST /api/v1/basket`
- `PUT /api/v1/basket`
- `DELETE /api/v1/basket`
- `GET /api/v1/order`
- `POST /api/v1/order`

### Контур поставщика

- `POST /api/v1/partner/update`
- `GET /api/v1/partner/state`
- `POST /api/v1/partner/state`
- `GET /api/v1/partner/orders`

## Локальный запуск

### Требования

- Python 3.10+
- виртуальное окружение `venv`

### Шаги запуска

```bash
python -m venv venv
venv\Scripts\activate
pip install -r requirements.txt
python manage.py migrate
python manage.py runserver
```

После запуска приложение будет доступно по адресу:

```text
http://127.0.0.1:8000/
```

Админка:

```text
http://127.0.0.1:8000/admin/
```

## Запуск в Docker

Для полного окружения с Redis и Celery можно использовать Docker Compose:

```bash
docker-compose up --build
```

Будут подняты:

- Django-приложение;
- Redis как брокер сообщений;
- Celery worker для фоновых задач.

Остановка контейнеров:

```bash
docker-compose down
```

## Фоновая обработка

Проект поддерживает два режима работы.

### С Redis и Celery

Импорт товаров и отправка email выполняются асинхронно.

### Без Redis

Проект продолжает работать без ошибок. Если брокер недоступен:

- импорт выполняется синхронно;
- письма отправляются синхронно;
- API остается доступным.

Это удобно для локальной разработки и демонстрации диплома без дополнительной инфраструктуры.
